What are Cookies?
Cookies or HTTP cookies are small key-value data blocks.
Servers send cookies in response and browser stores these cookies on your device.
These data blocks are stored to support various functionality. Like identifying user based on stored user-id in cookie.
Along with the key-value, cookies can also have other attributes, like expiry time.
SameSite is also an attribute which server can set on a cookie along with the key-value pair.
You can see cookies stored on your device using your browser.
If you are reading this article on your desktop's chrome Open Developer Console > Go to Application Tab > On the left panel, under Storage select Cookies > Select the domain (Website addresses) you want to see cookies for
One or more domain may have cookies stored on your device. Each cookie has many attributes like Name, Value, Domain, Expires, HttpOnly, Secure, SameSite etc.
In this article we will focus on SameSite
How does server sets the cookie?
When you access a website which wants to set a cookie. Within response, the backend server also returns Set-Cookie header.
Our browser sees Set-Cookie header and creates the cookie on our device.
Following is the example of a set-cookie header.
Set-Cookie: cookieKey='cookieValue'; Max-Age=2600000; Secure
JavaScript
Here, you can see along with key-value pair, we have two additional attributes Max-Age and Secure.
Max-Age, as name suggests, denotes the validity of the cookie. While Secure denotes this cookies will be sent only if the connection between server and client is on HTTPS
Cookie and Context
At a given time, your device might have cookies for a variety of domains not just the website you are currently visiting.
Because of this, cookie can be a first party cookie or a third party cookie. It is based on the context we are in.
Context is nothing but the domain of the current website we are using.
To decide if a cookie is first party cookie or a third party cookie compare the domain attribute set on cookie to website you are on.
first-party cookies: Cookies with domain = domain of current site.
third-party cookies: Cookies with domain ≠ domain of current site.
Cookie and browser's default behaviour
For every request made from your browser to any website, Browser checks if any cookies are stored for that website.
If yes it takes the decision whether it should send the cookie or not based on the attributes present on cookie + various other factors.
In early days. browsers won't do decision part. It would just check and send the cookie right away if it is present.
But it led to birth of Cross site request forgery (CSRF) attacks.
Why does the Default behaviour causes CSRF
Scenario: Free Pizza!
Before going to the pizza part. Let's assume, Your bank keeps the session data in a cookie.
Bank uses this data to identify the current user session. If this data is present and session is still valid you are not asked for username and password.
On a sunny good day, while you were using your internet banking, you get an email which says:
🍕
Awesome Pizza Shop opening near you soon! We are giving free pizza on our launch day which is coming weekend. All you have to do is register yourself. By clicking link below Register See you at the weekend!!
It seemed like a good idea and you clicked on the link.
❗
Always be cautious about clicking unknown links
It takes you to the website where you fill a form and submit.
Website says Thanks for registering! and then you get a text message
💸
You account xxxx13131 has been debited of $100
What?!!! 😱
Here is what happened.
Guess what? You got scammed. The hacker has hid a payment transfer link behind the free pizza website.
When you click this link it triggers a payment transfer link to your bank.
The browser checks if there is any cookie present for your bank's website.
It finds one and this cookie has your session information
Browser sends that cookie to the bank.
Bank sees there is an active session so it doesn't ask for the password.
Bank sees no problem and transfer the money.
❗
Lesson: Browsers should be careful while sending the cookie. Hence decision making step was included to prevent these kinds of attacks.
What is same site?
An attribute. Specifically designed to inform browser when to send the cookie.
It can have one of three values: Strict, Lax and None
When server returns following http header, it sets mycookie = 1 along with the SameSite to Strict
Set-Cookie: mycookie=1; SameSite=Strict
JavaScript
How do decide SameSite Value for your cookie?
The value of Same Site and the behaviour of browser whether to send the cookie or not depends upon two things
Origin Site of the request
Destination Site of the request
Strict, Lax and None
Strict: Send cookie when the Origin and Destination are domain of the cookie.
Lax: Send Cookie when Destination is the domain of the cookie + Navigation is a top level navigation i.e. address bar changes to Destination site. It reflects clearly that you are going to other website.
🍕
One point to remember for Lax is. If the top level navigation is POST then Lax cookie won't be sent. This we will discuss in detail.
None: Send Cookie in All cases.
🍕
For cookies with SameSite=None you must also specify the Secure attribute to ensure cookie is only transmitted on a secure channel. This is to ensure an extra layer of security. Set-Cookie: mycookie=1; SameSite=None; Secure
Default value for SameSite
SameSite attribute is must for a cookie.
If it is blank then Browser will treat it as SameSite=Lax
One of the main reasons for the change to SameSite=Lax as the default for cookies was to protect against Cross-Site Request Forgery (CSRF).
💡
If bank will have set SameSite=Lax on session cookie the free pizza attack would not work.
Why? Because scam website doesn't include a top level navigation to bank's website.
And the attacker can't trigger the default cookie sending behaviour by embedding target website behind a fake pizza shop website.
Chromium's Lax + Post Mitigation
Default Chromium behaviour is slightly less strict than normal SameSite=Lax behaviour
Chrome allows cookie sending for POST requests as well

User Cases when SameSite=None makes sense
Embedded content shared from third party website and embedded as <iframe>
Let say "Visit Counter" is a website which provide a widget to count visitor on your website.
All you have to do is embed its widget using <iframe>.
If you don't know what <iframe> is, It's an HTML tag to embed content from other website into yours.
Visit Counter does the counting using a cookie
Set-Cookie: count=1; SameSite=?
JavaScript
When user visits your site, this cookie will be set by visitcounter.com.
Since cookie is set by visitcounter.com and your website is different domain. Count cookie will be third party cookie.
Your website is third party context for count cookie.
Our goal is to ensure this cookie is sent to visitcounter.com every time the user visits our website.
What would be the appropriate value for SameSite here?
We can't keep Strict because start and destination is not visitcounter.com.
We can't keep Lax too. Why? Because when user visits our site it's not a top level navigation to visitcounter.com.
None? Yes! We have to keep it as None
"Unsafe" request and SameSite=Lax
"Unsafe" request : Here, the meaning of Unsafe is not any attack or something. But a term which was used to define POST requests.
Why Unsafe? Because POST request can change the data on the server side.
Let say, instead of widget of Visitor Counter. You Ask users to click on a link to visitcounter.com, which takes them to the website and record the count.
Yeah silly use case! But stay with me on this.
Here the navigation is a top level navigation. You are actually getting redirected to visitcounter.com.
so mywebsite.com → visitcounter.com with top level navigation? Sounds like Lax to me.
But as mentioned if the request type is POST. Then it's flagged as "Unsafe" hence even with Lax, cookie won't be sent.
So ? None it is.
Same site and Scheme.
Scheme simply means HTTP or HTTPS.
Before, HTTP or HTTPS versions of a same website were considered as same site.
But now they will be considered as cross-site. This is known as Schemeful SameSite
Why is this necessary?
Cookies are vulnerable when they are transmitted over less secured HTTP instead of HTTPS.
If a request navigates from HTTP → HTTPS, an attacked can tamper the cookie while it is in HTTP and tempared cookie will end up in HTTPS which is expected to be secured from tampering.
Schemeful SameSite is an extra layer to protect users from this attack.
Schemeful SameSite and browser's behaviour
Common cross-scheme scenarios
Request originates from one scheme and goes to other with top level navigation.
Strict is blocked because scheme change = different site.
Lax is allowed because its a top level navigation.
None is allowed when the destination scheme is secured because Secure is default attribute with None.
Loading subresources
A resource/widget is embedded in the page.
Strict is blocked because due to scheme change means different site.
Lax is blocked because its not a top level navigation i.e address in the address bar doesn't change.
None is allowed when the destination scheme is secured because Secure is default attribute with None.
POST form submit request
Top level navigation to different scheme with POST
Strict is blocked because due to scheme change means different site.
Lax is blocked because it's a top navigation but it's a POST.
None is allowed when the destination scheme is secured since Secure is default attribute with None.
Conclusion
Cookies are an important component of our internet.
With cookies and browser's default behaviour came CSRF.
SameSite helps browser to decide when it should send the cookie.
Understanding SameSite is an important part of using cookie correctly.
